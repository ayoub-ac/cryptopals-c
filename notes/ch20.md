# Challenge 20 — Break fixed-nonce CTR statistically

**Link:** https://cryptopals.com/sets/3/challenges/20

## What it does
60 plaintexts (rap lyrics) all encrypted with AES-128-CTR using the same key and the same nonce. Recover the keystream and decrypt them.

## Why it breaks
Same root cause as ch19: reusing a nonce in CTR is fatal. CTR is a stream cipher that XORs each plaintext with `keystream = AES(key, nonce || counter)`. If two messages share the keystream, you can attack them like a many-time pad.

## Strategy
1. Truncate every ciphertext to the length of the shortest one (so every column is full)
2. For each column `i`, all bytes were XORed with the same keystream byte → it's a single-byte XOR cipher
3. Run `crack_single_byte_xor` on each column → recover one keystream byte
4. XOR the keystream against each ciphertext to decrypt

## Difference vs ch19
Ch19 had short hand-picked sentences. Ch20 has 60 longer lyrics, which makes the statistical scoring much more reliable per column.

## Functions
- `load_plaintexts()` — read base64 lines from a file, decode to bytes
- `encrypt_all_fixed_nonce()` — encrypt every plaintext with the same key and nonce=0
- `recover_keystream_stat()` — column-by-column attack using single-byte XOR cracking

## Reuses from
- `aes_128_ctr()` from ch18
- `crack_single_byte_xor()` from ch03
- `base64_to_bytes()` from ch01

## Summary
Never reuse a nonce in CTR mode. With enough ciphertexts you don't even need known plaintext: english letter frequency alone is enough to recover the entire keystream column by column.
