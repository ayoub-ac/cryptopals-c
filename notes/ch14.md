# Challenge 14 — Byte-at-a-time ECB decryption (Harder)

**Link:** https://cryptopals.com/sets/2/challenges/14

## What it does
Same as ch12 but now the oracle adds a random prefix before our input: `ECB(prefix + input + secret)`. We still need to decrypt the secret.

## How it works
The random prefix messes up our block alignment from ch12. We need to figure out how long the prefix is first, then pad our input to align everything.

## Strategy
1. **Find prefix length**: send increasing amounts of identical bytes until two consecutive blocks in the ciphertext are the same. That tells us where the prefix ends.
2. **Align**: add enough bytes to push the prefix to a block boundary.
3. **Same attack as ch12**: byte-at-a-time, just offset by the prefix blocks.

## Finding the prefix length
Send 32+ identical bytes ('A'). With enough A's, at least two full blocks will be identical. The position of those blocks tells us where the prefix ends. Try adding 0, 1, 2... extra bytes until the identical blocks appear.

## Functions
- `ecb_oracle_harder()` — same as ch12 oracle but with random prefix
- `find_prefix_length()` — detects prefix size by looking for repeated blocks
- `decrypt_ecb_harder()` — the full attack with prefix handling

## Reuses from
- `aes_128_ecb_encrypt()` from ch07
- `base64_to_bytes()` from ch01
- Same byte-at-a-time logic from ch12

## Summary
Adding a random prefix doesn't stop the attack, it just adds one extra step. Once you know the prefix length, you can align your input and do the exact same chosen-plaintext attack.
