# Challenge 14 — Byte-at-a-time ECB Decryption (Harder)

**Link:** https://cryptopals.com/sets/2/challenges/14

## What it does
Same as ch12, but the oracle prepends a random prefix of unknown length: `ECB(prefix + input + secret)`. Recover the secret.

## Key insight
The prefix is the only new obstacle. Once we know its length and align past it, the attack is identical to ch12.

**Finding the prefix length:**
Send 32+ identical bytes (`AAAA...`). ECB will produce identical ciphertext blocks wherever the plaintext has two identical consecutive 16-byte blocks. Try padding values 0, 1, 2... until two consecutive identical blocks appear. The position of those blocks reveals where the prefix ends.

```
prefix = 5 bytes, pad = 11 extra A's:

block 0: [PPPPP AAAAAAAAAAA]  ← prefix fills exactly to boundary
block 1: [AAAAAAAAAAAAAAAA]   ← all A's
block 2: [AAAAAAAAAAAAAAAA]   ← all A's → block 1 == block 2 → FOUND

prefix_len = block_position * 16 - pad = 1*16 - 11 = 5
```

**Aligning past the prefix:**
Once we know prefix_len, calculate how many extra bytes are needed to push it to a block boundary: `align = (16 - prefix_len % 16) % 16`. From that point, ignore the first `(prefix_len + align) / 16` blocks — they're all prefix. Attack the remaining blocks exactly like ch12.

## Why it matters
A random prefix is a common real-world scenario (session tokens, fixed headers prepended to user data). This challenge shows that it doesn't stop the attack — it just requires one extra step to measure and neutralize the prefix. The byte-at-a-time attack is robust against unknown prefix lengths.

## Functions
- `ecb_oracle_harder()` — prepends random prefix before input + secret
- `find_prefix_length()` — force two identical blocks, read off prefix length
- `decrypt_ecb_harder()` — find prefix → align → same byte-at-a-time loop as ch12
