# Challenge 10 — Implement CBC Mode

**Link:** https://cryptopals.com/sets/2/challenges/10

## What it does
Implement AES-128-CBC from scratch using only raw AES-ECB per block. Cryptopals explicitly forbids using OpenSSL's built-in CBC — the goal is to understand the chaining manually.

## How CBC works

CBC (Cipher Block Chaining) adds a dependency between blocks to fix ECB's main weakness.

**Encryption formula:**
```
C[0] = AES_encrypt(P[0] XOR IV)
C[i] = AES_encrypt(P[i] XOR C[i-1])
```

**Decryption formula:**
```
P[0] = AES_decrypt(C[0]) XOR IV
P[i] = AES_decrypt(C[i]) XOR C[i-1]
```

The IV (initialization vector) acts as `C[-1]` for the first block. It should be random and unpredictable — a fixed IV weakens CBC significantly.

## The critical detail: order of operations

For **encryption**: XOR the plaintext with the previous ciphertext block FIRST, then pass the result to AES. Swap the order and you break the chaining.

For **decryption**: run AES_decrypt on the ciphertext block FIRST, then XOR with the previous ciphertext. Again, order matters.

Getting this backwards produces wrong output with no obvious error — the code compiles and runs, just decrypts garbage.

## Why CBC is better than ECB

Because each plaintext block is XORed with the previous ciphertext before encrypting, identical plaintext blocks produce completely different ciphertext. The same `AAAAAAAAAAAAAAAA` block encrypts differently every time depending on what came before it.

## Why this matters for ch16

The decryption formula `P[i] = AES_decrypt(C[i]) XOR C[i-1]` is the foundation of the bitflipping attack. It means modifying `C[i-1]` directly XORs into `P[i]` during decryption — with bit-precise control, no key needed.

## Functions
- `aes_128_cbc_encrypt()` — PKCS#7 pad → for each block: XOR with prev cipher → AES encrypt
- `aes_128_cbc_decrypt()` — for each block: AES decrypt → XOR with prev cipher → PKCS#7 unpad
