# Challenge 16 — CBC Bitflipping Attack

**Link:** https://cryptopals.com/sets/2/challenges/16

## What it does
An oracle encrypts user input inside a fixed cookie string with AES-CBC. A second function decrypts it and checks for `;admin=true;`. The oracle sanitizes input by stripping `;` and `=`, so we can't inject them directly. We forge a valid ciphertext that passes the admin check without knowing the key.

## The key formula

CBC decryption:
```
P[N] = AES_decrypt(C[N]) XOR C[N-1]
```

`C[N-1]` XORs byte-for-byte into `P[N]` at decryption time. This means: if we flip a bit in `C[N-1]`, the same bit flips in `P[N]`. We don't need the key — we only need to know the current plaintext value and what we want it to become.

To change byte at position `pos` from current value `cur` to target value `tgt`:
```
C[N-1][pos] ^= cur ^ tgt
```

## Structure of the encrypted buffer

The oracle builds: `prefix (32 bytes) + input + suffix`. Our input `"XadminXtrueX..."` lands in block 2 (bytes 32–47). The three `X` characters are at positions 0, 6, and 11 within that block — chosen because they align with `;`, `=`, `;` in `;admin=true;`.

To turn the `X`s into the target characters, we flip bytes in block 1 of the ciphertext (bytes 16–31), which XORs directly into block 2 during decryption:

```c
cipher[16 + 0]  ^= 'X' ^ ';'   // position 0:  X → ;
cipher[16 + 6]  ^= 'X' ^ '='   // position 6:  X → =
cipher[16 + 11] ^= 'X' ^ ';'   // position 11: X → ;
```

## What happens at decryption

- **Block 0** decrypts normally → `"comment1=cooking"`
- **Block 1** (the one we modified) decrypts to garbage — AES avalanche: one bit changed in the ciphertext scrambles all 16 bytes of that block's plaintext. This is acceptable; the server doesn't care about block 1.
- **Block 2** decrypts as: `AES_decrypt(C[2]) XOR C[1]_modified`. Since `C[2]` was untouched, `AES_decrypt(C[2])` produces the same internal value `K` as before. XORing with the modified `C[1]` shifts the three target positions from `X` to `;`, `=`, `;`.
- **Blocks 3–5** decrypt normally.

Final plaintext seen by the check function:
```
"comment1=cooking[16 bytes garbage];admin=true;;comment2=..."
                                    ^^^^^^^^^^^^
                                    found by strstr → admin = 1
```

## The lesson: encryption ≠ integrity

AES-CBC provides confidentiality — an attacker without the key cannot read the plaintext. But it provides no integrity guarantee: an attacker *with access to the ciphertext* can modify it in a controlled way and the server cannot detect the tampering.

The fix is **Encrypt-then-MAC**: compute a MAC (e.g. HMAC-SHA256) over the ciphertext, verify it before decrypting. Alternatively, use an **AEAD mode** like AES-GCM or ChaCha20-Poly1305, which combines encryption and authentication in a single operation. Either approach makes this attack impossible — any modification to the ciphertext invalidates the authentication tag.

## Functions
- `bitflip_encrypt()` — builds prefix + sanitized(input) + suffix, encrypts with AES-CBC
- `bitflip_check_admin()` — decrypts and searches for `;admin=true;`
- `cbc_bitflip_attack()` — three XOR lines on the ciphertext, no key required
