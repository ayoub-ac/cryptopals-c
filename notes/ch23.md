# Challenge 23 — Clone an MT19937 RNG from its Output

**Link:** https://cryptopals.com/sets/3/challenges/23

## What it does
Observe 624 consecutive outputs from an MT19937 generator. Reconstruct its full internal state. Predict all future outputs exactly.

## Why 624 outputs are enough

MT19937 keeps an internal state of exactly 624 `uint32_t` values (`mt[0..623]`). Each call to `mt_extract()` returns `temper(mt[index])` and advances the index. After 624 calls, all 624 state words have been output — one per extraction. If we can invert the tempering transform, we can recover `mt[i]` from each output and reconstruct the full state array.

## Inverting the tempering transform

The temper function applies four operations in sequence:
```c
y ^= (y >> 11)
y ^= (y << 7)  & 0x9D2C5680
y ^= (y << 15) & 0xEFC60000
y ^= (y >> 18)
```

To untemper, apply the inverse of each operation in **reverse order**.

**Inverting a right-shift XOR** (`y = x ^ (x >> s)`):
The top `s` bits of `y` equal the top `s` bits of `x` unchanged (XOR with 0). Use those to recover the next `s` bits, then the next, iteratively:
```
x = y                        // top s bits correct
x = y ^ (x >> s)             // top 2s bits correct
x = y ^ (x >> s)             // top 3s bits correct (converges in ceil(32/s) steps)
```
For `s=18`: 2 iterations sufficient (18+18=36 > 32).
For `s=11`: 3 iterations sufficient (11+11+11=33 > 32).

**Inverting a left-shift AND XOR** (`y = x ^ ((x << s) & mask)`):
Same idea from the other direction — bottom `s` bits of `y` are unchanged:
```
x = y
x = y ^ ((x << s) & mask)   // repeat ceil(32/s) times
```
For `s=15`: 3 iterations. For `s=7`: 5 iterations.

## The clone

After untempering 624 outputs, we have `mt[0..623]` — the exact state the victim generator was in after its last twist. Load that array into a fresh MT instance and set `index = 624`. On the next extraction, both the victim and the clone will call `generate()` (the twist) with the same state and produce identical results from that point forward.

## What would prevent this attack

If each tempered output were passed through a cryptographic hash function (e.g. SHA-256) before being returned, the output would reveal nothing about the internal state — hash functions are one-way. This is how cryptographically secure PRNGs (CSPRNGs) like Fortuna or /dev/urandom work internally.

## Functions
- `mt_untemper()` — invert all four tempering steps in reverse order
- `mt_clone_and_predict()` — tap 624 outputs → untemper → load state → predict next N values
