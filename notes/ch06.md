# Challenge 6 — Break Repeating-key XOR

**Link:** https://cryptopals.com/sets/1/challenges/6

## What it does
A file was encrypted with repeating-key XOR and then base64 encoded. Find the key and decrypt it. This is the hardest challenge in Set 1.

## How it works
Hamming distance counts how many bits differ between two byte buffers. If you compare blocks of ciphertext using the correct key length, the normalized hamming distance will be smaller than with wrong key lengths. Once you know the key length, you can split the ciphertext into columns where each column was XORed with the same byte — turning it into multiple single-byte XOR problems (ch03).

## Strategy
1. Read and decode the base64 file (using `read_base64_file` from ch01)
2. For key sizes 2 to 40, compute normalized hamming distance between blocks
3. The key size with the smallest distance is probably correct
4. Transpose the ciphertext into columns (one per key byte position)
5. Crack each column as single-byte XOR (reusing `crack_single_byte_xor` from ch03)
6. Combine the key bytes and decrypt with `repeating_key_xor` (ch05)

## Functions
- `hamming_distance()` — count differing bits between two buffers
- `guess_keysize()` — try keysizes 2-40, pick smallest normalized distance
- `break_repeating_key_xor()` — full attack: guess key length, transpose, crack, decrypt

## Reuses from
- `read_base64_file()` from ch01 — to read and decode the input file
- `crack_single_byte_xor()` from ch03 — to solve each transposed column
- `repeating_key_xor()` from ch05 — to decrypt with the found key

## Summary
This challenge ties together several previous ones. Hamming distance helps us figure out how long the key is — the idea is that blocks encrypted with the correct key length will be more similar to each other. Once we know the key length, we rearrange the ciphertext so that each column contains bytes XORed with the same key byte. That turns the whole thing into multiple ch03 problems. The recovered key was "Terminator X: Bring the noise" (29 bytes).
