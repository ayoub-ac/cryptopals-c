# Challenge 18 — AES-128-CTR (Counter mode)

**Link:** https://cryptopals.com/sets/3/challenges/18

## What it does
CTR mode turns AES (a block cipher) into a stream cipher. It generates a keystream by encrypting sequential counter blocks, then XORs that keystream with the data. Encryption and decryption are the same operation.

## How it works
For each 16-byte block of input:
1. Build a counter block: `nonce (8 bytes LE) || counter (8 bytes LE)`
2. Encrypt that counter block with AES-ECB to get 16 bytes of keystream
3. XOR the keystream with the input block
4. Increment the counter

The last block can be shorter than 16 bytes — we just use as many keystream bytes as we need.

## Why CTR is cool
- No padding needed (unlike ECB/CBC)
- Same function encrypts and decrypts (XOR is its own inverse)
- Can encrypt/decrypt blocks in any order (random access)
- Parallelizable since each block is independent

## Why CTR can be dangerous
- Never reuse a nonce with the same key. If you do, XORing two ciphertexts cancels the keystream and you get plaintext XOR plaintext, which is very breakable.

## Functions
- `aes_128_ctr()` — encrypt or decrypt data using AES-128 in CTR mode

## Reuses from
- Raw AES-ECB block encryption (same pattern as ch10's `ecb_block_encrypt`)

## Summary
CTR is my favorite mode so far. It's simple to implement — just encrypt a counter and XOR. No chaining between blocks, no padding. The tradeoff is that nonce reuse is catastrophic, which is exactly what the next challenges will exploit.
