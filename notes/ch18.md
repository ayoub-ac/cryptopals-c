# Challenge 18 — AES-128-CTR (Counter Mode)

**Link:** https://cryptopals.com/sets/3/challenges/18

## What it does
Implement AES-128-CTR mode and use it to decrypt a given base64 ciphertext.

## How CTR works

CTR turns AES (a block cipher that needs exactly 16 bytes) into a stream cipher that handles any length. It generates a keystream by encrypting a counter, then XORs it with the data:

```
keystream[i] = AES(key, counter_block[i])
ciphertext[i] = plaintext[i] XOR keystream[i]
```

The counter block format (as specified by Cryptopals, little-endian):
```
[ nonce (8 bytes, LE) | counter (8 bytes, LE) ]
  counter starts at 0 and increments by 1 per block
```

Each 16-byte counter block produces 16 bytes of keystream. For the last block, we only use as many bytes as needed — no padding required.

Since XOR is its own inverse, encryption and decryption are identical operations. The same `aes_128_ctr()` call works both ways.

## Properties

**Good:**
- No padding needed (unlike ECB/CBC)
- Encrypt and decrypt are the same function
- Random access: to read byte `N`, compute only counter block `N/16`, no need to process earlier blocks
- Fully parallelizable since counter blocks are independent

**Fatal weakness:**
Never reuse a nonce with the same key. If two plaintexts are encrypted with the same `(key, nonce)` pair, both share the same keystream:
```
cipher_A XOR cipher_B = plain_A XOR plain_B
```
The keystream cancels and the attacker gets plaintext XOR plaintext — directly breakable with frequency analysis. This is exactly what ch19 and ch20 exploit.

## Functions
- `aes_128_ctr()` — encrypt or decrypt any-length data using AES-128-CTR (same function for both directions)
