# Challenge 17 — The CBC Padding Oracle

**Link:** https://cryptopals.com/sets/3/challenges/17

## What it does
An oracle encrypts one of 10 secret strings with AES-CBC and gives us the ciphertext + IV. A second function decrypts arbitrary ciphertexts and leaks one bit: whether the PKCS#7 padding was valid. Using only that single-bit leak, recover the plaintext without the key.

## The key formula

CBC decryption, block by block:
```
P[N] = AES_decrypt(C[N]) XOR C[N-1]
```

Let `I[N] = AES_decrypt(C[N])`. This is the "intermediate state" — it only depends on the key and `C[N]`, not on the previous block. The real plaintext is `I[N] XOR C[N-1]`.

If we replace `C[N-1]` with any block `C'` of our choosing and ask the oracle to decrypt `C' || C[N]`, the oracle computes:
```
P' = I[N] XOR C'
```
and tells us whether `P'` ends with valid PKCS#7 padding. Since we control `C'` byte by byte, we can force specific values into `P'` and use the oracle as a one-bit yes/no on padding validity.

## The attack, byte by byte

We attack the last byte of a block first, then move left.

**Last byte (i = 15):**
We want `P'[15] = 0x01` (valid 1-byte padding). Iterate `g` from 0 to 255, setting `C'[15] = g`. Exactly one `g` produces `I[15] XOR g = 0x01`, so:
```
I[15] = g XOR 0x01
```

**Disambiguation:** the real plaintext block might already end in longer valid padding (e.g. `... 0x02 0x02`). In that case multiple values of `g` pass the oracle. To confirm we really found `0x01`, flip a bit in `C'[14]` and re-query: if padding is still valid, only byte 15 mattered, so pad is truly `0x01`.

**Next byte (i = 14):**
Now that we know `I[15]`, fix `C'[15] = I[15] XOR 0x02` so `P'[15] = 0x02`. Iterate `g` on `C'[14]` until the oracle accepts → that's the `g` where `P'[14] = 0x02`, giving us `I[14] = g XOR 0x02`.

**General step (i):**
Set `C'[j] = I[j] XOR pad_val` for `j = i+1..15` where `pad_val = 16 - i`. Then iterate `g` on `C'[i]` until the oracle validates `pad_val` bytes of padding.

After recovering all 16 bytes of `I[N]`, the real plaintext is:
```
P[N] = I[N] XOR C[N-1]   (use the real previous block, or the IV for block 0)
```

Repeat for every block. First block uses the IV as `C[N-1]`; blocks 1..n-1 use the preceding real ciphertext block.

## Why it works

The oracle was supposed to leak nothing, but it leaks one bit. That single bit — valid padding or not — is enough because we can frame the decryption problem as "is this byte equal to value X?" and repeat it until we've pinned down every byte. 256 tries per byte, 16 bytes per block, ~4K oracle calls per block. A 4-block message: ~16K queries, a second or two.

## The lesson

Never leak the reason a decryption failed. Treat "bad padding" and "bad MAC" identically — same error message, same timing, same behavior. Better: switch to **AEAD** modes (AES-GCM, ChaCha20-Poly1305) where authentication is built in and padding oracles are impossible by construction. For legacy CBC, always apply **Encrypt-then-MAC**: verify a MAC over the ciphertext before even attempting to decrypt. If the MAC fails, you never reach the padding check, so there's nothing to leak.

This attack broke real systems: ASP.NET (2010), Ruby on Rails, Java Server Faces, iMessage. It's one of the most important cryptographic attacks of the last 20 years because the leak looks harmless — "wrong padding" seems like a normal error — but it's a full plaintext recovery.

## Functions
- `padding_oracle_encrypt()` — picks a random plaintext, encrypts with a fresh IV
- `padding_oracle_check()` — decrypts and returns 1 if PKCS#7 padding is valid, 0 otherwise
- `padding_oracle_decrypt()` — runs the attack block by block using only the oracle
- `recover_intermediate()` (static) — the inner loop that pins down `I[N]` one byte at a time
